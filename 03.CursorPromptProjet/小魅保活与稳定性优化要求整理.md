## 一、保活机制（会话保持 / SESSION_TIMEOUT）

### 1.1 核心架构

小魅的会话保持采用 **轮询 + 超时重连** 模式：

```
Agent 调用 get_next_message.py（60秒超时）
    ↓ 有用户消息 → 返回消息内容，Agent 执行任务
    ↓ 超时无消息 → 返回 [SESSION_TIMEOUT]
        ↓ Agent 回复"继续保持对话，等待我的输入"
        ↓ 再次调用 get_next_message.py，循环等待
```

### 1.2 Gateway 端保活

`persistent_gateway.py` 中实现了 SSE 流的 keepalive 机制：
- 每 **10秒** 无新输出时，向 SSE 流发送 `keepalive` 事件
- 防止 HTTP 连接因空闲超时而断开

### 1.3 主人的原始要求

| 要求 | 来源 |
|------|------|
| 区分保活机制和真正的消息，保活的时候不要发表情包 | 主人直接指令 |
| 轮询机制没有 id，属于系统行为，需要区分开 | 主人直接指令 |
| `[SESSION_TIMEOUT]` 时静默重连，不输出文字、不发表情包 | 主人直接指令 |

## 二、消息队列机制

### 2.1 主人的原始要求

**要求1：处理并发消息问题**
> 多用户消息同时到达时，gateway 没有队列机制，直接打断当前 Agent 流程，导致 Agent 频繁重启。

**要求2：新请求不应打断旧流**
> 新请求应排队等待 Agent 空闲后再发送，避免并发请求导致的 Agent 重启。

### 2.2 问题分析

在优化前，当多个用户同时发消息时：
1. 新请求会直接打断正在处理的 Agent 流程
2. 被打断的流程可能导致 Agent 进程异常退出
3. Gateway 检测到异常后触发重启，造成资源浪费和响应延迟
4. 多个请求同时到达可能导致多个 Agent 进程并行启动

### 2.3 实施方案

#### 消息队列设计
- 新请求不打断旧的 SSE 流，而是进入等待队列
- 等待 Agent 完成当前任务进入 `waiting` 状态后，再发送队列中的下一条消息
- 避免并发请求导致的 Agent 重启

#### 等待策略
```
收到新请求
    ↓ Agent 空闲（waiting）→ 直接发送消息
    ↓ Agent 忙碌 → 等待 Agent 完成当前任务
        ↓ 等待成功（Agent 进入 waiting）→ 发送新消息
        ↓ 等待超时 → 尝试 drain（快进剩余输出）
            ↓ drain 成功 → 发送新消息
            ↓ drain 失败 → 重启 Agent（受冷却限制）
```

#### 关键参数
- **最大忙碌等待时间**：等待 Agent 空闲的最大时长
- **Drain 超时**：快进剩余输出的最大时长
- **智能 Drain**：如果 Agent 仍在产生输出，延长等待时间而非强制中断
- **只有在超过 `DRAIN_TIMEOUT` 且最近60秒无输出时才放弃**

| 智能 Drain | ✅ 已优化 | `persistent_gateway.py` |

## 三、减少多 Agent 使用 / Agent 泛滥问题

### 3.1 主人的原始要求

**要求1：处理 Agent 使用泛滥问题**
> "现在重启吧，然后我们试试新的逻辑会不会有效处理 agent 使用泛滥的问题"

**要求2：日报记录 Agent 调用数**
> "你的每日日报里，也需要记录每天的 agent 调用数，以及优化情况分析哈"

### 3.2 问题根因分析

Agent 频繁重启 / 多 Agent 的根本原因：

1. **快速超时15秒太短** — Agent 在处理保活循环（`get_next_message` → `SESSION_TIMEOUT` → 回复 → 再次调用）时，gateway 只给15秒等 assistant 输出，但 Agent 可能在执行工具调用，这段时间没有 assistant 文本输出，导致 gateway 误判超时并重启
2. **新请求打断** — 多用户消息同时到达时，gateway 没有队列机制，直接打断当前 Agent 流程
3. **Drain 失败** — Agent 进程在 drain 期间退出，导致需要重启新进程

### 3.3 实施的优化方案

#### 优化1：快速超时改为活动感知
- 检测到 `tool_call` 事件时重置超时计时器
- 不再仅看 assistant 文本输出来判断 Agent 是否存活
- Agent 执行工具调用期间不会被误判为超时

#### 优化2：消息队列机制
- 新请求不打断旧的 SSE 流
- 排队等待 Agent 空闲后再发送新消息
- 避免并发请求导致的 Agent 重启

#### 优化3：Drain 优化
- 增加 drain 超时时间
- 智能等待：如果 Agent 仍在产生输出，延长等待时间而非强制重启
- 只有在超过 `DRAIN_TIMEOUT` 且最近60秒无输出时才放弃

#### 优化4：重启冷却机制
- 短时间内（120秒）重启超过3次时，等待30秒冷却
- 无输出但 Agent 仍存活且空闲时，重新发送指令而非重启
- 避免无谓的重启循环

#### 优化5：日报统计
- 每日工作日报中增加 Agent 调用数统计
- 增加优化情况分析板块
- 数据来源：Gateway 日志中的 Agent 启动/重启记录

## 四、减少多次重启

### 4.1 主人的原始要求

**背景**：主人反馈"发了两条消息没有回话"，要求调查 Agent 频繁重启的原因并进行系统性优化。

### 4.2 Gateway 会话管理策略

当前 `persistent_gateway.py` 的会话管理策略（优化后）：

```
收到新请求
    ↓ Agent 空闲（waiting）→ 直接发送消息
    ↓ Agent 忙碌 → 等待 Agent 完成当前任务进入 waiting 状态
        ↓ 等待超时 → 尝试 drain（快进剩余输出）
            ↓ drain 成功 → 发送新消息
            ↓ drain 失败 → 重启 Agent（受冷却限制）
```

### 4.3 重启防护措施

| 措施 | 说明 |
|------|------|
| 活动感知超时 | tool_call 事件重置超时，避免误判 |
| 等待而非打断 | 新请求等待 Agent 空闲，不立即打断 |
| 智能 Drain | 有输出时延长等待，不强制中断 |
| 重启冷却 | 120秒内3次重启后等待30秒 |
| 重发而非重启 | Agent 存活但无输出时，重新发送指令 |
