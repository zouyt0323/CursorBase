## 核心内容摘要

### 第 1 页 - 封面
**Cursor Agent Skills 核心原理与工程推演**
- 副标题：打造高级 AI 助手的白板手记：文件结构、加载机制与控制法则
- 来源：NotebookLM

### 第 2 页 - Skill 的本质定义
**Klee One Skill 是面向特定领域的入职培训手册**
- 通用 AI 助手 + 入职培训手册 = 领域专家
- 四大核心能力：
  - **专业工作流**：特定领域的多步骤流程（例如：合同审查流程、CI/CD 部署）
  - **工具集成**：特定文件或 API 的操作指南（例如：.docx 处理、Playwright 测试）
  - **领域知识**：公司特定知识与业务逻辑（例如：数据库 Schema、品牌规范）
  - **捆绑资源**：直接可用的辅助资产（例如：PDF 旋转脚本、前端模板）

### 第 3 页 - 精简原则
**精简是第一原则，因为上下文窗口是极度稀缺的公共资源**
- 上下文窗口中包含：System Prompts、Other Skills Metadata、Chat History
- Skill 的生存空间极其有限
- **核心假设**：Agent 已经非常聪明，只添加它绝对不知道的信息。不要教它如何写基本的 Python 循环。

### 第 4 页 - 三级渐进式加载
**三级渐进式加载机制保护了宝贵的上下文空间**
- **第 1 级：元数据（Metadata）**
  - 触发条件：始终在上下文中
  - 用于判断是否触发此 Skill
  - 容量约束：约 100 词（name + description）
- **第 2 级：SKILL.md 正文**
  - 触发条件：仅当 Skill 被成功触发后加载
  - 包含核心工作流
  - 容量约束：< 5000 词
- **第 3 级：捆绑资源（Bundled Resources）**
  - 触发条件：完全按需加载
  - 脚本甚至可以直接执行而无需读入上下文
  - 容量约束：无物理大小限制

### 第 5 页 - 文件解剖学结构
**一个标准 Skill 的文件解剖学结构**
```
skill-name/
├── SKILL.md          → 必需的主文件。包含 YAML 前言与 Markdown 正文。
└── 捆绑资源/（可选）
    ├── scripts/      → 可执行代码（Python/Bash）。Agent 的工具箱。
    ├── references/   → 详细参考文档。按需读入上下文的资料库。
    └── assets/       → 输出资源。如 Logo、PPT 模板、字体等。
```

### 第 6 页 - 捆绑资源类型选择
**按需选择最合适的捆绑资源类型**

| 资源类型 | 何时使用 | 加载方式 | 示例 |
|----------|----------|----------|------|
| scripts/ | 需要极高确定性、可靠性，或避免重复编写相同代码时 | 直接在后台执行，不需读入上下文 | rotate_pdf.py |
| references/ | Agent 执行任务时确实需要参考的复杂文档或业务逻辑 | 根据任务进度，按需读入上下文 | api_docs.md, schema.md |
| assets/ | 最终输出成果中需要引用的静态文件 | 仅被复制或修改路径，不需读入上下文 | logo.png, template.pptx |

### 第 7 页 - SKILL.md 蓝图
**SKILL.md 的标准建筑蓝图与职责划分**
- **YAML Frontmatter（触发器）**—— 占 15%
  - 包含 name（Skill 标识符）与 description（触发条件和功能描述）
  - 只有这里的内容始终对 Agent 可见！所有"何时使用"的信息必须放在这里
- **Markdown 正文（操作指南）**—— 占 85%
  - 具体的任务执行步骤、API 参考和代码约束
  - 只有 Skill 触发后才加载
- **黄金法则：将正文严格控制在 500 行以内**

### 第 8 页 - Description 字段
**Description 字段是决定 Skill 是否被唤醒的 SEO 搜索引擎**
- 必须同时明确声明"做什么"与"什么时候用"
- 穷举用户可能使用的所有同义词（例如：'Word doc', 'word document', '.docx'）
- **高阶技巧：覆盖负面条件**
  - 明确告诉 Agent 什么时候不要用它
  - 明确的排除条件能极大降低相似 Skill 共存时的误触发率

### 第 9 页 - 代码示例优先
**代码示例的沟通效率远高于冗长的文字解释**
- 精简的代码示例 vs 冗长的文字解释：代码示例更高效
- **挑战每条信息**：
  - 写下每一句话前反问："Agent 真的需要这段解释才能工作吗？"
  - 剔除所有通用编程常识
- **500 行红线**：如果 SKILL.md 正文逼近 500 行，说明上下文正在被严重污染，必须启动内容分离

### 第 10 页 - 操作自由度
**根据任务的脆弱程度设定合适的操作自由度**
- **低自由度 / 悬崖窄桥**
  - 适用场景：操作脆弱易错（如直接编辑 OOXML 或 DXA 单位）
  - 控制手段：提供绝对具体的脚本，严格的代码模板和不可逾越的规则
- **高自由度 / 开阔田野**
  - 适用场景：多种方案均可，允许探索（如选择 Playwright 测试框架）
  - 控制手段：仅提供高层级的文字指导和验收标准，允许 Agent 自行变通决策

### 第 11 页 - 内容分离架构
**突破 500 行限制的渐进式内容分离架构**
- **高层指南 + 参考文件**：SKILL.md 仅作为概览与导航节点，按需挂载子文档
- **按领域平行组织**：主文件 → 按领域分子文件（如 finance.md, sales.md, product.md）
  - 问及销售指标时，Agent 会智能判断仅加载 sales.md
- **严格约束**：
  - 避免深层嵌套引用
  - 超过 100 行的参考文件必须在顶部添加目录
  - 绝对禁止信息重复

### 第 12 页 - 排版心理学
**运用排版心理学精准控制 Agent 的注意力分配**
- **WRONG**：绝不要手动插入 bullet 字符
- **CORRECT**：使用 numbering config
- 单独禁止某事效果很差。Agent 看到 ❌ 会产生强烈的避险倾向
- 必须同时展示导致错误的典型坏代码与正确的标准答案
- **CRITICAL 标签节制使用**：在注释中使用大写的 `// CRITICAL:` 标记极度脆弱的参数。整个 Skill 中最多只允许出现 5-8 个，滥用会导致注意力衰减
- **强制降维的步骤编号**：多步骤流程必须使用明确的编号，提供无法跳过的物理执行顺序锚点

### 第 13 页 - 结构化导航
**结构化导航优于冗长的文字流程说明**
- **Quick Reference 映射表**：在顶部放置表格，Agent 能在读取前 3 秒内找到正确的工具路径，无需扫描全文
- **ASCII 决策树替代文字 IF-ELSE**：
  - Agent 极度擅长解析树状结构
  - 决策树比文字段落节省约 60% 的 Token，且消除了分支遗漏的风险

### 第 14 页 - 安全沙箱
**将外部脚本作为受保护的执行黑盒与安全沙箱**
- **强制隔离源码**：明确指令不要阅读脚本源码！总是先运行 `--help`。100 行代码占 300 Token，而 `--help` 输出不到 50 Token
- **划定 Auto-repair 的能力边界**：清晰告知 Agent 自动修复机制能解决什么，以及不能解决什么（如 XML 嵌套错误），防止盲目重试
- **防御性资源清理**：教育 Agent 使用 finally 块。先优雅关闭 `process.terminate()`，超时后强制清理 `process.kill()`，彻底杜绝僵尸进程
