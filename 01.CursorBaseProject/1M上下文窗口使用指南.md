# Claude 1M 上下文窗口使用指南

> 适用于 Cursor IDE + Claude Opus 4.6 / Sonnet 4.6
>
> 最后更新：2026-03-15

---

## 概述

Claude Opus 4.6 和 Sonnet 4.6 支持 **100 万 token** 的上下文窗口（约 75 万个英文单词 / 150 万个中文字符）。这意味着你可以在一个会话中加载整个代码库、数百页文档或完整的 Agent 执行历史。

## 1M 上下文 vs 传统上下文的区别

| 维度 | 200K 上下文 | 1M 上下文 |
|------|------------|----------|
| 代码库 | 需要分模块加载 | 可加载整个中型项目 |
| 文档 | ~150 页 PDF | ~600 页 PDF（最多 600 页） |
| Agent 会话 | 频繁 compaction，丢失细节 | 长时间运行不压缩 |
| 跨文件理解 | 依赖 RAG 或手动选择文件 | 直接加载所有相关文件 |
| Token 效率 | 反复搜索同一信息 | 一次加载，多次引用 |

## 在 Cursor 中的最佳实践

### 1. 充分利用大上下文减少 Compaction

**Compaction** 是 Cursor 在上下文接近上限时自动压缩历史对话的机制。1M 上下文大幅减少了 compaction 频率。

**建议**：
- 使用 Opus 4.6 模型（Max/Team/Enterprise 用户默认 1M）
- 在长任务中保持同一会话，避免频繁开新对话
- 复杂调试任务在一个会话中完成搜索、分析和修复

### 2. 优化 @ 引用策略

| 场景 | 传统做法 | 1M 上下文做法 |
|------|----------|--------------|
| 修改跨多文件的功能 | 逐个 @file | 一次性 @folder 加载整个目录 |
| 理解项目架构 | 分多次询问 | 一次加载核心文件，一次性理解 |
| 代码审查 | 分 PR 的 diff 块 | 加载完整 diff + 相关源文件 |

### 3. 减少不必要的 RAG 依赖

1M 上下文意味着很多之前需要 RAG 检索的场景可以直接加载到上下文中：

| 场景 | 是否仍需 RAG |
|------|-------------|
| 项目内部文档（< 500 页） | 可直接加载，不需要 RAG |
| 外部知识库（持续更新） | 仍需要 RAG |
| 历史对话记录 | 1M 上下文内可保持，超出需要 Memory MCP |
| 大型代码库（> 100 万行） | 仍需要 RAG 或 @codebase |

### 4. Agent 工作流优化

```
传统流程（200K）:
  搜索 → compaction → 再搜索 → compaction → 丢失上下文 → 调试循环

1M 流程:
  搜索 → 分析 → 再搜索 → 聚合 → 修复 → 全部在一个窗口
```

**具体建议**：
- MCP 工具调用的完整 trace 保持在上下文中
- 多步骤任务（搜索 + 分析 + 修改 + 测试）在同一会话完成
- 减少 SubAgent 的使用频率（直接在主 Agent 中完成更多工作）

### 5. 媒体处理能力提升

| 类型 | 旧限制 | 新限制 |
|------|--------|--------|
| 图片 | 100 张/请求 | 600 张/请求 |
| PDF 页面 | 100 页/请求 | 600 页/请求 |

**应用场景**：
- 一次性分析整个 UI 设计稿（多屏截图）
- 加载完整的技术规格文档（PDF）
- 批量处理架构图和流程图

## Token 消耗估算

| 内容类型 | 大约 Token 数 |
|----------|-------------|
| 1 行代码 | ~15 token |
| 1 页 A4 文档 | ~500 token |
| 1000 行代码文件 | ~15,000 token |
| 100 个文件的中型项目 | ~150 万 token（需选择性加载） |
| 一次完整 Agent 会话（含工具调用） | ~50,000-200,000 token |

**1M token 可以容纳**：
- ~66,000 行代码
- ~2,000 页文档
- ~5-20 个完整的 Agent 工作会话

## Cursor 配置建议

### 模型选择

```
复杂任务（架构设计、大规模重构）: Opus 4.6（1M 上下文）
日常编码（补全、小修改）: Sonnet 4.6（1M 上下文，更快更便宜）
简单任务（格式化、注释）: Haiku（快速响应）
```

### 减少 Compaction 的技巧

1. **避免在对话中粘贴大量无关内容** — 只加载需要的文件
2. **使用 @file 而非复制粘贴** — Cursor 会更高效地处理引用
3. **定期开新会话** — 当任务完成后开新对话，而非在旧对话中堆积
4. **利用 cursor-usage 插件** — 监控 token 消耗，了解每次请求的效率

### 与 Memory MCP 配合

1M 上下文解决了**单次会话**内的记忆问题，但**跨会话**记忆仍需 Memory MCP：

```
单次会话内: 1M 上下文自动保持
跨会话: Memory MCP 持久化关键信息
跨项目: RAGFlow 知识库检索
```

## 成本优化

| 策略 | 说明 |
|------|------|
| 按需加载 | 不要一次性加载整个项目，按任务需要加载 |
| 选择合适模型 | 简单任务用 Sonnet/Haiku，复杂任务用 Opus |
| 监控 token 使用 | 使用 cursor-usage 插件跟踪消耗 |
| 复用会话 | 相关任务在同一会话中完成，避免重复加载上下文 |
