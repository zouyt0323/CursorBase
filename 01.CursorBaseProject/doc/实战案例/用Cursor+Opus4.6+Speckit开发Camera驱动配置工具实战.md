## 一、项目背景：为什么要做这个工具？

### 1.1 痛点分析

高通平台 Camera 驱动开发中，工程师面临一个繁琐但又不可避免的环节——手写 XML 驱动文件。

每颗新 Sensor 上线都需要编写一份 XML 驱动配置文件，涵盖 I2C 从机地址、电源序列、初始化寄存器、分辨率信息等上百个参数。这些 XML 文件必须严格遵循高通定义的 XSD Schema，格式要求极其苛刻：

- 一个标签写错，编译就报错
- 一个 0x 忘加，寄存器值就对不上
- 一个字段遗漏，设备就点不亮

传统工作流如下：

> FAE 提供 Datasheet/寄存器配置 → 工程师逐行手抄到 XML → 反复编译调试 → 推送设备验证

一颗新 Sensor 的驱动配置，熟练工程师也需要 1-2 天，新手可能需要 **一周甚至更久**。

### 1.2 目标：让写驱动 XML 变成填表单

**核心思路**：既然 XML 的结构由 XSD 定义，那完全可以自动解析 XSD，生成可视化表单，让工程师在网页上"填表"而不是"写代码"。

**更进一步**：有了 AI 加持，连表单都不用手动填——AI 分析 Datasheet 和寄存器配置文件后自动填写。

## 二、为什么选择 AI 辅助开发？

### 2.1 挑战评估

这个项目涉及多个技术领域：

| 技术领域 | 具体内容 | 难度 |
|----------|----------|------|
| 前端开发 | React + TypeScript + Tailwind CSS | 中 |
| 后端开发 | FastAPI + Python | 中 |
| XML/XSD 解析 | lxml 解析复杂嵌套 Schema | 高 |
| AI 集成 | LLM API 调用、流式输出、结果解析 | 高 |
| ADB 调试 | 跨平台 ADB 通信、日志实时显示 | 中 |
| 编译系统 | 调用高通 ParameterParser 工具链 | 中 |

如果纯手工开发，即使是全栈工程师，保守估计也需要 **3-6 个月**。

### 2.2 AI 开发工具选型

最终选择了 **Cursor IDE + Opus4.6** 的组合：

- **Cursor**：基于 VS Code 的 AI 增强 IDE，支持上下文感知代码生成
- **Claude（Anthropic）**：代码理解和生成能力强，支持超长上下文（200K token）
- **开发模式**：对话式开发 —— 描述需求 → AI 生成代码 → 验证 → 迭代

### 2.3 核心方法论：「需求驱动 + 渐进式开发」Speckit 首次运用在完整项目

```
用户需求（自然语言）
    ↓
AI 理解需求 → 分析现有代码 → 生成实现方案
    ↓
代码生成 → 自动编译 → 自动测试
    ↓
用户验证 → 反馈问题 → 下一轮迭代
```

**关键原则**：

1. **先讨论方案再写代码** —— 每个新功能先确认技术方案
2. **最小改动原则** —— 每次只改必要的部分
3. **参考现有实现** —— 复用已有的代码风格和模式
4. **渐进式迭代** —— 每轮只解决 3-5 个问题

## 三、AI 开发全过程详解

### 阶段一：基础框架搭建（第 1 周）

**目标**：从零搭建前后端框架，实现最基础的 XSD → 表单功能。

**AI 对话示例**：

```
我: 我需要一个高通 Camera 驱动的可视化配置工具。前端 React，后端 FastAPI。
    核心功能是解析 XSD Schema 文件，自动生成可填写的表单，让工程师
    在网页上配置驱动参数。支持 SM8650 和 SM8750 平台。

AI: [分析 XSD 结构] → [生成项目骨架] → [实现 XSD 解析器] → [生成表单渲染组件]
```

**AI 实际完成的工作**：

1. **项目初始化**：一次性生成了完整的项目结构
   - `frontend/` — React + TypeScript + Vite + Tailwind
   - `backend/` — FastAPI + lxml + pydantic
   - 配置文件（package.json、requirements.txt、tsconfig.json 等）

2. **XSD 解析引擎**：AI 分析了 31 个 XSD 文件的结构，编写了递归解析器
   - 支持 `xs:complexType`、`xs:simpleType`、`xs:sequence`、`xs:choice` 等
   - 处理 `range` 属性、`enumeration` 约束、嵌套引用

3. **动态表单渲染**：根据解析结果自动生成 UI
   - 支持文本输入、下拉选择、数组添加/删除
   - 字段分组、折叠/展开

**遇到的问题**：

> **问题**：initSettings 只加载了部分参数，很多寄存器没有显示
>
> **AI 定位**：XSD 解析器对 `maxOccurs="unbounded"` 的处理不完整
>
> **修复**：AI 修改了递归解析逻辑，正确处理无限重复元素

---

### 阶段二：核心编辑功能完善（第 2 周）

**迭代内容**：这一阶段的需求主要来自实际使用反馈，每轮包含 3-5 个问题：

| 轮次 | 问题 | AI 解决方案 |
|------|------|-------------|
| 1 | 界面不够紧凑 | 实现多列布局（2/3/4列可切换） |
| 2 | 缺少批量导入功能 | 开发寄存器粘贴解析器，支持多种格式 |
| 3 | 生成的 XML 与源文件差异大 | AI 反复对比 imx258 驱动，逐项修复 |
| 4 | 注释丢失、16进制变10进制 | 修改 XML 序列化逻辑，保留注释和 0x 前缀 |
| 5 | 复制字段后修改会同步 | 深拷贝 bug 修复 |

**批量导入功能的设计**：

AI 分析了工程师的实际工作习惯，设计了三种导入格式：

```
格式 1（简洁）：
2648 ff
2649 ff

格式 2（含从机地址）：
6c 2648 ff
6c 2649 ff

格式 3（完整）：
0x6c 0x112 0xa 2 1 WRITE 0
```

**一个典型的 Bug 修复案例**：

```
我: 生成的驱动里 <dt>47</dt>，但源文件是 <dt>0x2F</dt>，
    registerAddr 也从 0x112 变成了 274

AI: [分析原因] 字段值在解析时被隐式转为十进制。
    [修复方案] 在 XML 序列化时检测原始值是否有 0x 前缀，
    有则保留十六进制格式；对 `range` 属性内的值也做同样处理。
    [执行] 修改 xml_generator.py 的 serialize_value() 函数
    [验证] 加载 imx258 → 不修改 → 生成 → 与源文件 diff → 0 差异
```

### 阶段三：AI 智能功能开发（第 3-4 周）

这是整个项目最核心、最有技术含量的部分。

#### 3.1 AI 智能填写

**需求**：工程师上传 FAE 提供的 Datasheet 或寄存器配置文件（TXT/Excel），AI 自动分析并填写驱动。

**演进过程**：

| 版本 | 方案 | 问题 |
|------|------|------|
| v1 | 内置规则解析 | 只能处理固定格式，不够智能 |
| v2 | LLM 分析 + 内置规则填写 | 分析结果无法准确对应字段 |
| v3 | LLM 分析 + 发送 XSD + 示例驱动 | 大模型理解驱动结构，输出 XML |
| v4（最终） | 流式输出 + 实时进度 + 极简TXT | 稳定、专业、用户体验好 |

**最终方案架构**：

```
用户上传配置文件
    ↓
后端读取文件内容（支持 TXT/Excel/PDF，含密码保护）
    ↓
构造 Prompt：配置文件 + XSD Schema + 示例驱动(s5kjd1sp)
    ↓
调用 LLM（Claude sonnet-4.6）
    ↓
流式返回：实时显示分析进度
    ↓
解析 LLM 返回的 XML 驱动 + 极简 TXT
    ↓
一键应用到页面驱动字段
```

**Prompt 工程的关键**：

发给 LLM 的内容包括：
1. 用户配置文件（TXT/Excel 内容）
2. 对应平台的 XSD Schema（定义了合法的 XML 结构）
3. 一个完整的示例驱动（让 LLM 理解最终输出格式）
4. 明确的输出要求：
   - 输出完整的驱动 XML（可直接编译）
   - 输出极简 TXT（方便用户二次编辑）
   - 输出分析报告（每个字段的值及证据来源）

#### 3.2 AI 智能验证

**需求**：工程师完成驱动配置后，上传 Datasheet 让 AI 检查配置是否正确。

```
用户上传 Datasheet + 驱动 XML
    ↓
LLM 对比分析
    ↓
输出验证报告：
  ✅ 正确：I2C 地址 0x6C 与 Datasheet 一致
  ⚠️ 警告：寄存器 0x0100 的值建议为 0x01
  ❌ 错误：电源序列缺少 DOVDD
```

### 阶段四：编译推送与调试系统（第 4-5 周）

#### 4.1 一键编译

高通驱动 XML 需要通过 ParameterParser 工具编译为 BIN 文件才能使用。

**AI 解决的核心难题**：

```
问题：用户在任意目录生成 XML，但 XML 内部引用的 XSD 路径是相对路径，
      导致编译器找不到 Schema 文件。

AI 方案：编译前自动修改 XML 内 xsi:noNamespaceSchemaLocation，
         替换为正确的绝对路径，编译后恢复原值。
```

#### 4.2 ADB 在线调试

**需求**：工程师推送 BIN 后，需要查看 Camera 内核日志来验证驱动是否工作。

AI 开发了完整的调试系统：
- 实时 dmesg 日志显示（WebSocket 推送）
- 按模块过滤（cam-sensor / cam-isp / cam-cpas / cam-cci 等）
- 关键操作按钮（杀 Provider、I2C 调试开关、HW 调试开关等）
- 全屏日志模式（点击展开，隐藏左侧编辑区）
- 日志着色（Error 红色、Warning 黄色、不同模块不同颜色）

**演进过程**：

```
v1: 只在右边小区域显示日志，不能动态刷新
    ↓ 用户反馈：很难看，不实用
v2: 增加全屏模式，WebSocket 实时推送
    ↓ 用户反馈：杀 Provider 不工作，关闭相机不正常
v3: 修复命令执行，增加命令历史显示
    ↓ 用户反馈：需要模块过滤，需要颜色区分
v4（最终）: 完整的调试面板，10+ 个快捷操作按钮
```

### 阶段五：局域网部署与用户体验优化（第 5-6 周）

#### 5.1 局域网共享

**目标**：部署一台服务器，全组通过浏览器使用。

AI 解决的问题：
- **前后端分离部署**：nginx 反向代理 + FastAPI 后端

## 六、关键经验与总结

### 6.1 AI 开发擅长的领域

- **框架搭建**：AI 可以秒级生成完整的项目骨架（前后端、配置、工具链）
- **重复性工作**：如为 8 个驱动模块实现相同的编辑/创建功能
- **Bug 定位与修复**：描述现象 → AI 分析可能原因 → 给出修复方案
- **API 集成**：AI 熟悉主流库和框架的 API，能快速实现集成
- **UI/UX 开发**：给出需求描述后，AI 能生成高质量的 React 组件

### 6.2 AI 开发需要注意的地方

- **需要人工验证**：AI 生成的代码不一定完全正确，每次改动都需要测试
- **渐进式迭代更有效**：一次给太多需求，AI 容易遗漏或冲突；分批次效果更好
- **上下文管理很重要**：项目越大，需要越精确地告诉 AI 应该关注哪些文件
- **领域知识仍不可替代**：AI 不了解高通 Camera 驱动的具体业务逻辑，需要开发者引导
- **方案讨论优先于编码**：先让 AI 出方案，人工确认后再实施，避免返工

### 6.3 高效使用 AI 的方法论

```
第一步：清晰描述需求 + 提供上下文（相关文件、错误信息、示例）
    ↓
第二步：AI 出方案，人工审核（而不是直接让 AI 写代码）
    ↓
第三步：确认方案后，AI 编码 + 自动编译验证
    ↓
第四步：人工测试，记录问题（附截图/日志）
    ↓
第五步：反馈问题，进入下一轮迭代
    ↓
每 3-5 轮做一次全功能测试，防止回归
```

### 6.4 对 Camera Driver 工程师的启示

- **AI 不会取代你，但会让你效率翻倍**
- **学会"描述需求"** 比"写代码"更重要
- 把重复性工作交给 AI，把精力放在驱动调试和图像质量优化上
- 积累 Skill 和 Prompt：把常用的工作流沉淀为 AI 可复用的模式
