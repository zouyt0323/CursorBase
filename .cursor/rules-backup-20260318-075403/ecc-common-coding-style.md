# 编码风格

## 不可变性（关键）

始终创建新对象，绝不就地修改已有对象：

```
// 伪代码
错误：modify(original, field, value) → 就地修改原对象
正确：update(original, field, value) → 返回带变更的新副本
```

理由：不可变数据避免隐藏副作用，便于调试，支持安全并发。

## 文件组织

多小文件优于少大文件：
- 高内聚、低耦合
- 典型 200–400 行，最大 800 行
- 从大模块中抽取工具函数
- 按功能/领域组织，而非按类型

## 错误处理

全面处理错误：
- 在每一层显式处理错误
- 面向用户的代码提供友好错误信息
- 服务端记录详细错误上下文
- 绝不静默吞掉错误

## 输入验证

在系统边界进行验证：
- 处理前验证所有用户输入
- 优先使用基于 schema 的验证
- 快速失败并给出清晰错误信息
- 不信任外部数据（API 响应、用户输入、文件内容）

## 代码质量检查清单

完成前确认：
- [ ] 代码可读、命名清晰
- [ ] 函数短小（<50 行）
- [ ] 文件职责单一（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 有正确错误处理
- [ ] 无硬编码（使用常量或配置）
- [ ] 无就地修改（使用不可变模式）
